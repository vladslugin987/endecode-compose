# Endecode Compose: анализ проекта

## Что это за проект

`endecode-compose` — desktop-приложение на Kotlin + Compose Multiplatform для:
- внедрения и чтения watermark-меток;
- добавления видимого текста на изображения;
- batch-обработки папок и файлов.

По текущему состоянию это локальное GUI-приложение без серверной части и без БД.

## Технологический стек

- Kotlin Multiplatform: `1.9.20`
- Compose Multiplatform: `1.5.11`
- Coroutines: `1.7.3`
- AndroidX Lifecycle: `2.6.2`
- OpenCV (через `org.openpnp:opencv`): `4.7.0-0`
- Logback: `1.4.14`
- Gradle Wrapper: `8.9`
- Целевые пакеты: `DMG` (macOS), `MSI` (Windows)

## Структура проекта

- `composeApp/` — основной модуль приложения.
  - `src/desktopMain/.../Main.kt` — точка входа и создание окна.
  - `ui/` — экраны, диалоги и UI-компоненты.
  - `viewmodels/` — orchestration операций и состояния UI.
  - `utils/` — работа с watermark, batch-процессами, изображениями, файлами.
  - `data/PreferencesManager.kt` — локальные пользовательские настройки.
- `gradle/libs.versions.toml` — централизованные версии зависимостей.
- `.github/workflows/build.yml` — CI-сборка на GitHub Actions.
- `README.md`, `UI_REDESIGN_README.md`, `COMPILATION_FIXES.md` — документация и заметки.

## Как все работает (по слоям)

1. **UI-слой**  
   Главный экран и диалоги собирают входные параметры пользователя (пути, режимы, опции).

2. **ViewModel-слой**  
   `HomeViewModel` управляет жизненным циклом операций (`encrypt`, `decrypt`, batch-задачи и т.д.), прогрессом и логами.

3. **Utils/сервисный слой**
   - `EncodingUtils` — кодирование/декодирование строк + watermark-маркеры.
   - `WatermarkUtils` — добавление/извлечение/удаление watermark на уровне байтового хвоста файлов.
   - `BatchUtils` — пакетное копирование, watermark-пайплайн, zip и доп. операции.
   - `ImageUtils` — визуальный watermark на изображениях через OpenCV.

4. **Локальное хранение настроек**  
   `PreferencesManager` пишет свойства в `~/.endecode/endecode.properties`.

## Сборка и запуск

- Локально:
  - `./gradlew run` (macOS/Linux)
  - `gradlew.bat run` (Windows)
- Пакетирование:
  - `./gradlew packageDmg`
  - `./gradlew packageMsi`

CI (`.github/workflows/build.yml`) собирает проект в matrix на `macos-latest` и `windows-latest`, загружает артефакты `dmg/msi`.

## Ключевые наблюдения и риски

1. **В репозитории отсутствует `gradle-wrapper.jar`**  
   Есть `gradle/wrapper/gradle-wrapper.properties`, но без jar запуск wrapper может быть невоспроизводим на чистой машине.

2. **Термин “encryption” не равен криптографии**  
   В реализации используется Caesar-сдвиг + watermark-метки, это не защищенное шифрование.

3. **Риск для бинарных файлов в части потоков кодирования**  
   Некоторые операции читают/пишут содержимое как UTF-8 текст, что может быть небезопасно для бинарных форматов.

4. **Несогласованности UI и фактической логики**  
   В отдельных местах UI обещает поведение (например, backup), которое не всегда отражено в реализации.

5. **Незавершенные части UI/функций**  
   Есть TODO-ветки и потенциально неиспользуемые компоненты, которые стоит либо подключить, либо удалить.

6. **Потенциально чувствительные значения в plaintext**  
   Токены/поля авторизации сохраняются в properties-файле пользователя без шифрования.

- Свести в одно место контракт UI <-> backend-логики и закрыть расхождения.

## Быстрый итог

Проект выглядит как desktop-инструмент watermark/batch-обработки, уже пригодный для дальнейшей доработки, но требующий технической стабилизации вокруг сборки, точности терминов и безопасности/корректности обработки файлов.
